# 일본 콜센터 운영 구조

## 제품이 해야 하는 일

한 통화 세션에는 방향이 명확한 두 오디오 채널이 있습니다.

```text
고객 수신 음성 ─→ 외국어 STT ─→ 일본어 번역 ─→ 상담원 화면

상담원 마이크 ─→ 일본어 STT ─→ 고객 언어 번역 ─→ TTS ─→ 고객 송신 음성
```

현재 우선순위는 노트북 마이크와 스피커만 사용하는 독립형 왕복 통역기를 완성하는 것입니다. 이 단계에서 음성인식·번역·TTS·되먹임 차단·장시간 안정성을 먼저 검증합니다. PBX 또는 소프트폰의 수신/송신 오디오 직접 연결은 코어가 안정화된 뒤 진행합니다.

## 필요한 통화 연동

연동 방식은 사용하는 전화 시스템에 따라 달라집니다.

- 브라우저 WebRTC 콜센터: 웹앱의 수신 MediaStream과 송신 MediaStream 사이에 직접 연결
- Windows 소프트폰: 가상 오디오 장치 또는 해당 제품 SDK 사용
- SIP/PBX: 상담원별 RTP 미디어 프록시 또는 PBX 녹음/미디어 포크 API 사용
- 물리 전화기: USB 오디오 인터페이스가 수신/송신을 분리 제공해야 함

최종 구현 전에 PBX/소프트폰 제품명, 상담원 수, 동시 통화 수, 녹음 정책을 확정해야 합니다. 이 정보 없이 특정 오디오 연결 코드를 만드는 것은 납품 가능한 통합이 아니라 데모에 불과합니다.

## 운영 안전 조건

- 번역 엔진 장애가 원래 통화를 끊어서는 안 됨
- 상담원별 모델 상태·대화 언어·자막 기록 완전 격리
- TTS는 고객 송신 채널에만 주입하고 상담원 마이크로 되먹임되지 않게 함
- 낮은 STT/언어감지 신뢰도와 지원하지 않는 언어를 상담원에게 즉시 표시
- 계좌번호·전화번호·주소 같은 숫자는 원문과 번역을 함께 표시
- 자동 TTS 전송 전에 번역문 확인/취소 모드를 운영 정책으로 선택
- 고객에게 AI 통역 사용을 고지할지 법무·개인정보 담당자와 결정
- 로그 보존기간, 암호화, 접근권한, 삭제와 감사 추적 정의

## 서버 배치

한 PC에서 한 상담원이 쓰는 초기 버전은 로컬 모델이 가장 단순합니다. 상담원이 많아지면 PC마다 수 GB 모델을 배포하는 방식과 사내 GPU/CPU 추론 서버를 비교해야 합니다. 사내 서버도 외부 API 비용은 없지만 동시 통화 수에 맞는 용량 계획과 장애 이중화가 필요합니다.

Vercel은 고객 음성을 처리하는 위치로 사용하지 않습니다. 소개·설치 파일 배포에는 쓸 수 있지만, 통화 미디어와 모델 추론은 상담원 PC 또는 회사 내부 서버에 남겨야 합니다.

## 수용 테스트 예시

각 언어와 통화 환경에서 최소 다음을 측정합니다.

- 영어·한국어·스페인어·중국어 각각 100문장 STT 정확도
- 각 방향 번역의 상담원/원어민 사람 평가
- 고객 발화 종료부터 일본어 표시까지 p50/p95 지연
- 상담원 발화 종료부터 고객이 TTS를 듣기 시작할 때까지 p50/p95 지연
- 8시간 연속 실행, 100통 연속 세션, 네트워크/오디오 장치 재연결
- 고객과 상담원이 겹쳐 말할 때 채널 분리 및 문장 순서
- 모델 오류·디스크 부족·장치 분리 시 통화 우회와 복구
